Isso é a pista definitiva! O que você descreveu (funciona se navegar, mas some se der F5) confirma que não é um erro de salvamento, o dado está no banco.

O problema é um Conflito de Filtros de Data e Timezone (Fuso Horário).

Aqui está o que está acontecendo "nos bastidores" para causar esse comportamento bizarro:

O "Fantasma" do Fuso Horário: Você lança a venda hoje (03/01). O sistema salva ela como 00:00 UTC. No Brasil, isso equivale a Ontem às 21:00 (02/01).

O F5 (Atualizar Página): Quando você recarrega a página, o sistema aplica o filtro padrão: "Hoje". Como para o sistema a venda foi "ontem às 21h", ela é escondida.

A Navegação (Ir para Fornecedores e voltar): Ao navegar pelo menu, é muito provável que o seu sistema esteja limpando o filtro de data da memória (Estado Global), exibindo "Todas as Transações" ou "Mês Atual" ao invés de "Hoje". Por isso ela aparece!

Para resolver isso, precisamos garantir que a venda seja salva no meio do dia, para que ela nunca caia no dia anterior por causa de fuso horário.

"O comportamento do bug mudou e agora está claro: A transação é salva, mas desaparece quando dou F5 na página de transações, pois o filtro padrão é 'Hoje' e a transação provavelmente está ficando com a data de 'Ontem' devido à conversão de Fuso Horário (UTC vs Local). Ela só aparece quando navego entre telas porque isso deve estar resetando o filtro de data.

Preciso que você faça duas correções urgentes no arquivo NewSaleDialog.js (ou onde a venda é criada):

Forçar Hora Segura: Ao enviar a data da venda (saleDate e paymentDate), não envie apenas a data pura ou com T00:00:00Z. Force o horário para o meio-dia para evitar que o fuso horário (UTC-3) jogue a venda para o dia anterior.

Use algo como: saleDate: new Date(data.sale_date + 'T12:00:00').toISOString()

Resetar Filtros: No onSuccess, verifique se existe alguma função de contexto para limpar os filtros de data da tela de listagem, ou garanta que a invalidação do React Query force um refetch considerando a nova data."
